# 「学伴系统」初期开发策划案  
（面向大学生团队的轻量化学习激励+AI辅助系统）  


## 一、项目概述  
### 1. 项目背景  
当代大学生常面临“学习动力不足、目标模糊、缺乏反馈”等问题。参考小说中“系统任务-属性成长”的激励逻辑，结合AI辅助学习能力，打造一款轻量化工具，通过“自定义任务→完成获得属性提升→系统币兑换资源”的闭环，让学习过程更有目标感和成就感。  

### 2. 核心定位  
**“个性化学习激励助手”**：以游戏化元素（属性、任务、系统币）为激励载体，以AI辅助（答疑、专项指导）为实用工具，核心是让用户“愿意学、学得清、有反馈”。  

### 3. 目标用户  
初期聚焦大学生群体（尤其是需要自主规划学习的学生），支持自定义学习方向（如考研、考公、语言学习、专业课程等）。  


## 二、核心功能模块（初期MVP版）  
### 1. 用户系统  
- **基础功能**：注册/登录（支持学号/邮箱注册，密码加密存储）、个人中心（展示属性面板、任务进度、系统币余额）。  
- **核心价值**：建立用户专属数据空间，记录学习轨迹。  

### 2. 属性系统（核心特色）  
- **自定义属性**：用户可创建个性化属性（如“高数熟练度”“Python编程”“雅思词汇量”），设置初始值（0-100）。  
- **属性成长**：完成关联任务后，通过“手动自评”或“AI建议”获得属性点（如“完成3小时高数刷题，+5熟练度”），属性值实时展示（进度条/数值）。  
- **简化设计**：初期无需复杂算法，属性值仅作为学习成果的可视化象征，不直接影响任务难度（降低开发复杂度）。  

### 3. 任务系统  
- **任务创建**：用户自定义任务名称（如“背50个雅思单词”）、目标描述、关联属性、预计耗时、奖励（系统币数量）。  
- **任务流程**：创建→标记“进行中”→完成后提交（可上传证明，如笔记照片/打卡截图）→获得属性点+系统币。  
- **评估方式**：初期以“用户自评”为主（选择完成度，系统推荐属性点），后期加入AI简易评估（如识别笔记关键词，辅助判断完成质量）。  

### 4. 经济系统（激励闭环）  
- **系统币获取**：完成任务固定奖励（如简单任务5币，复杂任务20币），暂不开放其他获取途径（避免逻辑复杂）。  
- **兑换功能**：  
  - 学习资料：团队预先上传的免费资源（如期末复习提纲、网课链接、电子书），兑换需10-50币；  
  - AI服务次数：兑换“AI答疑”（单次5币）或“专项指导”（如“高数解题思路”，单次10币），每日兑换上限5次。  

### 5. AI辅助系统（实用工具）  
- **基础答疑**：调用公开AI接口（如豆包API），支持自然语言提问（如“如何理解微积分中的极限？”），基于通用知识回答。  
- **专项指导（RAG简化版）**：绑定1-2个高频学科知识库（如高数公式库、英语语法库，初期手动录入TXT文本），用户提问时，系统先检索知识库相关内容，再让AI基于该内容回答（减少错误）。  
- **初期限制**：不做复杂多轮对话，单次提问→单次回答，优先保证稳定性。  


## 三、技术方案  
### 1. 技术栈选型（低门槛+易上手）  
| 模块         | 技术选择                          | 理由说明                                  |  
|--------------|-----------------------------------|-------------------------------------------|  
| 前端         | Vue3 + Vite + Element Plus        | 组件丰富、文档友好，适合快速搭建界面       |  
| 后端         | Python + FastAPI                  | 语法简单，自带API文档，便于前后端对接      |  
| 数据库       | SQLite（初期）                    | 无需部署，文件型数据库，适合本地测试       |  
| AI接口       | 豆包API（优先）/GPT-3.5           | 免费额度高，调用简单，降低初期成本         |  
| RAG实现      | 关键词检索（初期）→ LangChain（后期） | 初期用Python字符串匹配，后期升级向量检索   |  
| 部署         | 本地测试→阿里云学生机（后期上线） | 初期团队内部测试，降低服务器成本           |  

### 2. 系统架构（简化版）  
```  
用户层 → 前端界面（Vue3）  
        ↓ ↑  
API层 → 后端接口（FastAPI）  
        ↓ ↑  
服务层 ├─ 用户服务（注册/登录/数据管理）  
        ├─ 任务&属性服务（任务CRUD/属性计算）  
        ├─ 经济服务（系统币发放/兑换）  
        └─ AI服务（API调用/知识库检索）  
        ↓ ↑  
数据层 → SQLite数据库（用户表/任务表/属性表/知识库表）  
```  


## 四、团队分工（4人团队）  
### 成员A：前端开发（2-3周）  
- 负责页面开发：登录页、个人面板（属性+任务展示）、任务创建/编辑页、商店兑换页；  
- 实现前端与后端的API交互（Axios），确保数据实时渲染（如任务完成后属性值即时更新）；  
- 优化UI：参考“游戏系统面板”设计，突出属性成长的视觉反馈（如进度条变色、数值跳动）。  

### 成员B：后端开发（2-3周）  
- 设计数据库表结构（用户表、属性表、任务表、知识库表、系统币记录表）；  
- 开发核心API：用户注册/登录（密码哈希存储）、任务CRUD、属性值更新、系统币兑换逻辑；  
- 对接数据库（SQLAlchemy），确保数据增删改查稳定，编写接口文档（供前端调用）。  

### 成员C：AI与知识库模块（3-4周）  
- 对接AI接口：申请豆包API，编写调用函数（接收前端问题→返回AI回答），实现“次数限制”（兑换后扣减次数）；  
- 搭建简易RAG：手动录入首批知识库（如50条高数基础题、50条英语语法点），用Python关键词匹配实现检索；  
- 辅助属性评估：编写简单逻辑（如“任务耗时1小时以上，推荐属性点+3~5”），供用户参考。  

### 成员D：产品设计与测试（全程）  
- 输出产品原型（Figma线框图），明确页面流程（如“创建任务→完成→领奖励→兑换AI”）；  
- 协调前后端开发，确认接口逻辑（如任务完成后如何触发属性和币的更新）；  
- 负责测试：验证全流程通顺性（如注册后能否创建属性、兑换AI能否正常调用），收集BUG并推动修复；  
- 准备初期资源：整理可兑换的学习资料（如团队专业课笔记、公开网课链接），录入系统。

## 六、初期简化策略（降低难度）  
1. **属性评估**：不用AI深度分析，用户自评+系统推荐值（基于任务耗时/描述）；  
2. **RAG检索**：跳过向量库，用Python字符串匹配（`in`关键字）实现简单检索；  
3. **系统币逻辑**：仅与任务完成挂钩，不做签到、分享等复杂获取方式；  
4. **AI功能**：限制单次问答，不支持多轮对话，优先保证调用成功率。  


## 七、风险与应对  
| 风险点               | 应对措施                                  |  
|----------------------|-------------------------------------------|  
| 技术难点（如RAG效果差） | 初期用关键词检索保底，后期引入LangChain优化 |  
| 数据量不足（属性评估不准） | 依赖用户自评，积累数据后再训练简单规则模型  |  
| 开发进度滞后          | 优先保证核心闭环（任务→属性→币），砍掉非必要功能 |  


## 八、预期成果  
1. 可运行的初期版本：实现“属性自定义→任务管理→奖励兑换→AI辅助”全流程；  
2. 团队能力提升：掌握前后端开发、AI接口调用、产品设计的基础逻辑；  
3. 验证用户需求：通过内部试用，判断“游戏化+AI”模式是否能提升学习动力，为后续迭代提供方向。  


**核心原则**：先做出“能用的最小版本”，再基于实际使用反馈迭代，避免因追求完美而停滞。帮我重新优化一下然后写成md然后就是不要那么明显的学伴系统不只是学生而且我更想保留小说中那种系统金手指的感觉，我更想用langchain当然云端agent也行吧
